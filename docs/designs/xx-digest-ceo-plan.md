# CEO Plan: xx digest — Complete Learning System

## Vision

### 10x Check
xx becomes a personal intelligence briefing system. Not just filtering tweets but bridging what you're building and what the world is producing. Context that compounds (TODO.md, project docs, planning docs). Action routing (adopt = issue, avoid = flag, copy = save). The briefing evolves weekly. The behavior change (days without opening X) IS the metric.

### Platonic Ideal
Open terminal in the morning. Three items waiting. Each one actionable, each one referencing your specific code. Over a week, the digest gets sharper. The product learns what you care about from every interaction. The user feels: "This tool knows what I'm building better than I do."

### Landscape Position
- No competitor combines production-behavior signals (git context) with consumption-behavior signals (keep/discard feedback). This is the eureka.
- Twigest (closest competitor) is web-based, generic, no work-context personalization.
- The two-signal-source architecture is rare and defensible.

## Product Principles (from WIRING-PLAN + Karpathy autoresearch)

1. **Not adding more work** — the feedback loop is zero-friction. Using the product IS the eval. No "rate this" prompts.
2. **Self-improve always** — the agent autonomously evolves its scoring rules based on accumulated feedback. The user never touches the prompt.
3. **Recovery by re-polling** — re-read context each time (git, feedback signals). No complex state restoration.
4. **Not rigid** — light harness. Model improvements naturally improve the product. Don't over-constrain.
5. **Progressive disclosure** — setup wizard reveals complexity gradually. Each phase builds on the last.
6. **Close loops** — every digest feels complete. Recover an item = close the loop on that signal.
7. **No manual config editing** — the setup wizard handles all configuration through interactive questions, not YAML editing. The wizard is the UI for config.

### Autoresearch Model Applied
The three autoresearch primitives map to xx:
- **Editable asset** = LLM system prompt + preference rules
- **Scalar metric** = signal rate: (explicit_keep + recover) / (explicit_keep + explicit_discard + recover). Higher = better digest. Implicit "accepted digest" signals excluded from denominator.
- **Time-boxed cycle** = each digest run IS a cycle
- **program.md** = the user's keep/discard actions program the agent

The agent reads feedback, rewrites preference rules autonomously, and the next digest run is the experiment. Self-improving with zero manual intervention.

## Approach Selected

**Approach C: Complete Learning System**
- Digest + setup wizard + inline feedback loop
- All three serve one integrated learning loop
- Setup teaches the product, digest filters the feed, feedback loop captures signals, agent self-improves

## Architecture Decisions

### Feedback Loop: Hybrid Few-shot → Distilled Rules
- Phase 1 (< 20 signals): Include 3-5 recent feedback examples as few-shot in the LLM prompt
- Phase 2 (20+ signals): Sub-agent distills accumulated signals into preference rules in the system prompt
- Crossover at 20 signals is when pattern recognition has enough data to be meaningful

### Self-Improvement Trigger: Signal-Count Based
- Distillation runs automatically after every 10 new signals
- Not on every digest run, not on a fixed schedule — data-driven
- Visible to the user: spinner + summary of what changed
- Sub-agent architecture: distillation agent reads feedback.jsonl, writes preference rules

### Inline Feedback UX: Inline After Digest
- After digest renders, optional prompt: "See filtered items? [y/N]" (defaults No)
- If yes: show discarded items with toggle to recover
- Every keep/discard/recover action logged to ~/.xxcli/feedback.jsonl
- One continuous flow, context is fresh

### Rich Work Context: Wizard-Populated
- Read well-known files by default (README.md, CLAUDE.md, TODO.md)
- Setup wizard scans repo, identifies project docs, asks user to confirm
- Additional context files stored in config.yaml but populated by wizard, never manual editing
- Graceful fallback if files don't exist

## Data Schemas

### feedback.jsonl (one record per user action)
```json
{
  "ts": "2026-04-04T09:15:00Z",
  "type": "keep|discard|recover|accepted_digest",
  "tweet_id": "1234567890",
  "score": 8,
  "classification": "adopt|avoid|copy|skip",
  "digest_run_id": "uuid",
  "context_repo": "~/Code/xxcli"
}
```
- `accepted_digest`: logged once per run when user declines to review filtered items. Contains no tweet_id. Lists shown item IDs in an `items_shown` array.
- Signal count for distillation trigger: count records where type is keep, discard, or recover (NOT accepted_digest).

### preference_rules.json (output of distillation sub-agent)
```json
{
  "updated_at": "2026-04-04T09:15:00Z",
  "signal_count_at_distillation": 20,
  "rules": [
    "Prioritize: developer tooling, agent architecture, CLI patterns",
    "Deprioritize: hype tweets, reaction-only content, meme accounts",
    "The user values actionability over novelty"
  ],
  "few_shot_examples": [
    {"tweet_summary": "...", "action": "keep", "reason": "..."},
    {"tweet_summary": "...", "action": "discard", "reason": "..."}
  ]
}
```

### Streak state (in config.yaml)
```yaml
streak:
  last_digest_run: "2026-04-04T09:00:00Z"
  consecutive_days: 5
  total_signals_surfaced_this_week: 12
```
"Days without opening X" = consecutive_days since first digest run. Resets if a day passes with no digest run. Aspirational framing, not surveillance.

## Error Handling

| Failure | Behavior | User sees |
|---------|----------|-----------|
| Missing OPENAI_API_KEY | Fail fast before any API call | `Error: No OPENAI_API_KEY found. Set it from your ChatGPT/Codex subscription.` |
| OpenAI API 429/500/timeout | 1 retry with 3s backoff. If retry fails, abort. | `Error: OpenAI API failed after retry. Try again in a few minutes.` |
| OpenAI returns malformed output | Pydantic validation catches it. 1 retry. | `Error: Scoring failed. Try again or run with --debug.` |
| Twitter API failure | Reuse `_handle_api_error` from cli.py | Existing error messages + rate limit hint |
| Corrupt/missing feedback.jsonl | Treat as empty. Recreate on next signal. | No visible error. Digest runs without preferences. |
| Distillation sub-agent fails | Skip distillation, use existing cached rules | `Warning: Preference update skipped. Using previous preferences.` |
| --repo path not a git repo | On first run: trigger wizard. On subsequent: inline warning. | `⚠ No work context — run from a git repo or use --repo PATH` |

## Sub-Agent Architecture

Two OpenAI Agents SDK agents defined in `llm.py`:

**1. Scoring Agent** (main)
- Model: gpt-4.1-mini (or gpt-5.4-mini when available)
- Input: tweets JSON + work context string + preference rules/few-shot examples
- Output: Pydantic `DigestResult(items: list[DigestItem])` via structured output
- Batching: all tweets (up to 100) in a single call. ~10K input tokens, well within context.

**2. Distillation Agent** (sub-agent, triggered every 10 new signals)
- Model: same as scoring agent
- Input: last 50 feedback signals from feedback.jsonl + current preference_rules.json
- Output: Pydantic `PreferenceRules(rules: list[str], few_shot_examples: list[Example])`
- System prompt: "You are analyzing a user's tweet curation behavior. Summarize their preferences as concise scoring rules. Reference specific topics, content types, and patterns."
- Writes output to `~/.xxcli/preference_rules.json`

## `xx why <id>` Specification

Caches digest results to `~/.xxcli/last_digest.json` on each run (all scored items, not just filtered). `xx why <tweet_id>` reads from cache and displays:
- The tweet text and author
- Relevance score and classification
- The explanation from the scoring agent
- Which work context matched (git commits, files, deps that correlated)
- Current preference rules that influenced scoring
If cache is stale (> 24h) or missing, re-runs scoring for that single tweet.

## Setup Wizard: Directory Scan Spec

Phase 1 scans these paths (macOS): `~/Code/`, `~/Projects/`, `~/Developer/`, `~/repos/`, `~/src/`, cwd. Depth: 1 level (direct children only). Identifies git repos by presence of `.git/` directory. Presents numbered list. User picks or enters custom path. If no dev directories found, prompt for manual path entry.

## Scoring Threshold

The `score >= 7` filter is a v1 default, not a permanent constant. The distillation sub-agent can recommend threshold adjustments in preference_rules.json (e.g., `"threshold": 6` if the user consistently recovers items scoring 5-6). This keeps the "not rigid" principle: the threshold can evolve with the user's preferences.

## Prerequisites

- Twitter API v2 Free Tier: home timeline endpoint is available. Rate limited to 15 requests/15 min. Sufficient for daily digest (1 call per run). The `referenced_tweets` expansion works on free tier for tweets returned in the response, no tier upgrade needed.
- OpenAI API key from ChatGPT/Codex subscription. No separate billing.
- Python 3.10+, pip install with openai-agents + pyyaml added to pyproject.toml.

## Data Flow

```
USER RUNS: xx digest
     │
     ├─── Check: 10+ new signals since last distillation?
     │    YES → Sub-agent distills feedback.jsonl → updated preference rules
     │          Show: "⏳ Evolving preferences... ✓ Prioritizing [X], filtering [Y]"
     │    NO  → Use cached preference rules
     │
     ├─── Pull timeline (client.py)
     ├─── Build work context (context.py: git + wizard-configured project docs)
     ├─── Build prompt: tweets + context + preferences (few-shot or distilled rules)
     ├─── Score tweets (llm.py: OpenAI Agents SDK, Pydantic structured output)
     ├─── Filter score >= 7, sort desc, truncate to --count
     │
     ├─── Render digest (format.py: header, items with tags + confidence dots, streak)
     │
     ├─── Prompt: "See filtered items? [y/N]"
     │    YES → Show discarded items, toggle to recover
     │          Log all keep/discard/recover to feedback.jsonl
     │    NO  → Log implicit "accepted digest as-is" signal
     │
     └─── Done. Feedback accumulates for next distillation trigger.

FIRST RUN: xx digest (or xx setup)
     │
     ├─── Phase 1: Pick repo (scan dev directories, present list)
     ├─── Phase 2: Scan for project docs ("I found these files — use for context?")
     ├─── Phase 3: Calibrate on real tweets (10-15 tweets, Useful/Maybe/Skip/Noise)
     │    ├─── First "Useful" → aha moment: "Got it. Prioritizing [topic] for [repo]"
     │    └─── Save calibration as initial feedback signals
     ├─── Phase 4: First digest with calibrated context
     └─── Save all to ~/.xxcli/config.yaml (wizard-populated)
```
