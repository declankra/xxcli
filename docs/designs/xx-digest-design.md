# Design: xx digest — Context-Aware Feed Digest

Generated by /office-hours on 2026-03-27

## Problem Statement

The user is a developer who uses X/Twitter to stay current on AI, tech, software, and business. The browser experience is addicting — the scroll pulls you in even when you opened X with a specific intent. The goal is to consume X content intentionally, not reactively, while protecting deep focus time.

The current xxcli (v0.1) is a feed reader that strips the browser chrome but doesn't solve the core problem: you still have to read every tweet and decide what matters. The signal-to-noise ratio of a raw timeline is low.

## What Makes This Cool

Every AI digest tool in the market personalizes from consumption behavior — what you click, what you read, how long you dwell. xx digest personalizes from production behavior — what you're actually building. Your recent commits, your repo structure, your dependencies, your README. That's a fundamentally stronger relevance signal for developers.

The demo line: "3 items worth your attention. 97% of your timeline was ignored."

The behavior change: "I haven't opened X in 3 days. My agent told me about a new framework relevant to what I shipped yesterday."

## Constraints

- **X API Free Tier.** Home timeline reads are available but rate-limited. The digest must work within free tier limits (currently 15 requests per 15-minute window for home timeline). A once-daily digest needs one batch pull. **Contingency:** If home timeline is revoked or further restricted on the free tier, fall back to search/recent endpoint or user-curated list endpoints. Reduced relevance but maintained functionality.
- **LLM: OpenAI only via ChatGPT/Codex subscription.** AI functionality is only supported through the user's existing ChatGPT subscription (Plus, Pro, or Codex). User authenticates with their `OPENAI_API_KEY` from their ChatGPT account — no separate API billing needed. Built on the **OpenAI Agents SDK** (`openai-agents`). Default model: `gpt-5.4-mini` (fast, cost-efficient, sufficient for relevance scoring). No other LLM providers supported.
- **Python ecosystem.** Must fit the existing stack: Python 3.10+, Click, Rich, Tweepy. No runtime framework changes.
- **CLI-native delivery.** Output goes to the terminal via Rich formatting. No web UI, no email, no push notifications in v1.
- **Single-user tool.** No multi-tenancy, no auth, no server. Runs on the user's machine.

## Premises

1. The X API free tier is sufficient for pulling enough tweets to build a meaningful digest (confirmed: home timeline endpoint available on free tier).
2. An LLM can reliably distinguish signal from noise in a developer's X feed when given work context.
3. The user will actually use a scheduled digest instead of opening X in the browser — the digest must be satisfying enough to kill the itch.
4. Work-context personalization (reading repos) is worth the complexity in v1 — this is the differentiator, not a nice-to-have.

## Cross-Model Perspective

Codex performed an independent cold read of the session context. Key findings:

- **Coolest unconsidered version:** `xx scout` — a standing brief that categorizes signals as `adopt` (start using), `avoid` (invalidates your approach), or `copy` (adjacent builders doing something relevant). This taxonomy was incorporated into the recommended approach.
- **Key insight:** The line "I haven't opened X in 3 days" is the product. Not "better Twitter." Not "AI summaries." It's: staying off the feed while still getting the one thing that mattered.
- **Existing tool:** gitingest (github.com/coderamp-labs/gitingest) turns a repo into prompt-friendly context. Gets 50% of the work-context problem solved. The other 50% is tweet scheduling, dedup, relevance ranking, and the "why this matters" explanation layer.
- **Weekend build:** `xx digest --repo ~/work/project --since 24h` — prints top 5-10 items with one-line summary, relevance explanation, direct link, and confidence score.

Claude agreed with the Codex build spec almost entirely. The adopt/avoid/copy taxonomy was a genuine addition that improves actionability of the output.

## Approaches Considered

### Approach A: Minimal Viable Digest
Pull home timeline, pass raw tweets to an LLM, get a top-N summary. No repo context, no relevance scoring. Generic summarization only.
- Effort: S
- Pros: Ships in a weekend, proves behavior-change bet, simple to debug
- Cons: No work-context personalization, not differentiated from existing digest tools
- Completeness: 5/10

### Approach B: Context-Aware Digest (CHOSEN)
Pull home timeline + build work context from git log, README, dependencies → LLM ranks tweets by relevance to current work → output top-N items with "why this matters" explanation and adopt/avoid/copy classification.
- Effort: M
- Pros: The eureka — production-behavior personalization. Sharp demo. Differentiated.
- Cons: Prompt engineering iteration needed. Context window management for large repos.
- Completeness: 8/10

### Approach C: Digest + Interaction Loop
Everything in B, plus `xx ask` to follow up on digest items with repo-aware Q&A. Goes from "interesting" to "applied" in one flow.
- Effort: L
- Pros: Full v1 vision
- Cons: Scope creep, two new features at once, conversation state management
- Completeness: 9/10

## Recommended Approach

**Approach B: Context-Aware Digest.**

### Architecture

```
xx digest [--repo PATH] [--since PERIOD] [--count N]
                |
                v
    +-------------------+
    |  1. Pull tweets   |  client.py: get_home_timeline()
    |     (last 24h)    |  max 100 tweets per API call
    +-------------------+
                |
                v
    +-------------------+
    |  2. Build work    |  NEW: context.py
    |     context       |  - git log --oneline -20
    +-------------------+  - README.md contents (truncated)
                |          - requirements.txt / pyproject.toml deps
                |          - Recently changed file paths
                v
    +-------------------+
    |  3. Agent ranking  |  NEW: digest.py + llm.py
    |     & scoring     |  - OpenAI Agents SDK agent w/ gpt-5.4-mini
    +-------------------+  - Score relevance 0-10
                |          - Classify: adopt / avoid / copy / skip
                |          - Generate "why this matters" for top items
                v
    +-------------------+
    |  4. Format &      |  format.py (extended)
    |     display       |  - Rich terminal output
    +-------------------+  - Summary header: "N items / M ignored"
                           - Each item: tweet, author, tag, explanation
                           - Tweet ID for follow-up actions (like, reply)
```

### New Files

- `src/xxcli/context.py` — Work context builder. Reads git log, README, deps from a target repo. Returns a structured context string suitable for LLM prompting.
- `src/xxcli/digest.py` — Digest engine. Takes tweets + work context, calls LLM API, returns ranked/classified results.
- `src/xxcli/llm.py` — Digest agent built on the OpenAI Agents SDK (`openai-agents`). Defines a single `Agent` with the system prompt as `instructions`, a `score_tweets` function tool for structured output, and `gpt-5.4-mini` as the model. Reads `OPENAI_API_KEY` from environment (provided by the user's ChatGPT/Codex subscription). The Agents SDK gives us automatic schema generation from Python type hints, built-in tracing for debugging prompt quality, and a clean path to multi-agent workflows later (e.g., separate context-builder and scorer agents).

### Modified Files

- `src/xxcli/client.py` — No changes needed for `--since`. The free tier home timeline endpoint does not support server-side `start_time` filtering — that requires Basic tier ($100/mo). Instead, `digest.py` pulls the max 100 tweets and filters by `created_at` timestamp client-side in Python before passing to the LLM. This keeps the tool free-tier compatible.
- `src/xxcli/cli.py` — Add `xx digest` command with `--repo`, `--since`, `--count` options.
- `src/xxcli/format.py` — Add `print_digest()` function for Rich-formatted digest output.
- `pyproject.toml` — Add `openai-agents` as a dependency (includes `openai` SDK as a transitive dep).

### CLI Interface

```bash
# Basic: digest from your home timeline, scored against current directory
xx digest

# Specify which repo to use as context
xx digest --repo ~/Code/my-project

# Control time window and result count
xx digest --since 24h --count 5

# Inspect the full LLM reasoning, all scores, timing, and token usage
xx digest --debug
```

**Flag details:**

- `--repo PATH` — Resolution order: CLI flag > `config.yaml default_repo` > current working directory. If the resolved path is not a git repo, the digest runs without work context and prints a warning: "Not a git repo — running without work context."
- `--since PERIOD` — Accepted formats: `Nh` (hours), `Nd` (days), `Nw` (weeks), or ISO-8601 date (`2026-03-26`). Default: `24h`. Parsed to a UTC `datetime` and used for **client-side filtering** — the API pulls up to 100 recent tweets, then Python filters by `created_at` before passing to the LLM. If the API returns fewer tweets than expected for the requested window, the digest proceeds with available tweets and the header stats reflect the actual count scanned.
- `--count N` — Maximum number of digest items to surface. Default: 5. The LLM returns all tweets scoring >= 7. `digest.py` sorts by score descending and truncates to `--count` client-side. This is more robust than asking the LLM to respect a count constraint.
- `--debug` — (from /plan-design-review, inspired by Linear's Agent Interaction Guidelines) Show the full LLM prompt, all tweet scores (including skipped), timing per phase (pull/context/score), and token usage. Renders in `[dim]` above the normal digest output. Essential for prompt iteration (Next Steps step 6) and debugging unexpected results. Follows Linear AIG principle: "underlying reasoning, tool calls, prompts, and decision logic should be inspectable."

### Agent Interaction Principles (from Linear AIG, incorporated via /plan-design-review)

The digest is an AI agent acting on the user's behalf. These principles from [Linear's Agent Interaction Guidelines](https://linear.app/developers/aig) are baked into the design:

1. **Immediate feedback** — Three-phase loading spinner provides unobtrusive status at every step. "Silence creates user uncertainty."
2. **State communication** — Each spinner phase tells the user exactly what's happening: pulling, building context, scoring. User understands the operation "at a glance."
3. **Inspectability** — `--debug` flag exposes full reasoning: prompt, all scores, timing, tokens. The "Why" explanation is a summary; `--debug` is the full picture.
4. **Human accountability** — The digest recommends (adopt/avoid/copy) but never acts. The user decides via `xx like`, `xx reply`, or doing nothing. "Final responsibility always remains with a human."
5. **Platform-native** — The digest operates through the same CLI patterns as `xx feed`, `xx post`, `xx like`. No special interface. Same terminal, same interaction model.

### LLM Prompt Design (core of the product)

**Draft prompt template** (starting point, iterate in step 6):

```
SYSTEM:
You are a relevance filter for a developer's Twitter/X feed. You score
tweets by how relevant they are to the developer's CURRENT work, not
general interest. Be ruthless — most tweets score 0-3.

Filter rules:
- Hype/reaction tweets that just amplify news without adding perspective: score 0.
- Tweets that are merely "interesting" but not actionable: score 1-3.
- The acid test: "does this change what the developer builds or how they think?"
- SERENDIPITY ALLOWANCE: For up to 1 item, you may score a tweet 7+ if it
  is genuinely novel/surprising and connects to the developer's domain in a
  non-obvious way, even if not directly about their current repo. Creativity
  comes from association. Do not let relevance filtering become a black hole.

USER:
## What the developer is building

Project: {repo_name}
Languages/frameworks: {deps_summary}
Recent commits:
{git_log_last_20}

README excerpt:
{readme_first_200_lines}

Recently changed files:
{changed_file_paths}

## Tweets to evaluate

{tweets_json_array}

## Instructions

For each tweet, return a JSON object with:
- tweet_id (string): the tweet's ID
- relevance_score (integer 0-10): how relevant to the developer's current work
- classification (string): one of "adopt", "avoid", "copy", "skip"
  - adopt: tool, library, technique, or API the developer should start using
  - avoid: something that invalidates or weakens the developer's current approach
  - copy: pattern, technique, or approach from adjacent builders that maps to this project
  - skip: not relevant to current work (score < 7)
- explanation (string): one sentence explaining WHY this matters to the developer's
  current work. Reference specific files, deps, or recent commits. Empty string if skip.

Return ONLY a JSON object: {"items": [...]}
Only include tweets with relevance_score >= 7 in the items array.
If no tweets score >= 7, return {"items": []}.
```

**Expected response JSON schema:**
```json
{
  "items": [
    {
      "tweet_id": "string",
      "relevance_score": 7,
      "classification": "adopt",
      "explanation": "You're using Tweepy in client.py — this new endpoint would simplify your home timeline pulls."
    }
  ]
}
```

**Token budget estimate:** 100 tweets at ~280 chars = ~28K chars (~7K tokens). README 200 lines = ~8K chars (~2K tokens). Git log 20 commits = ~2K chars (~500 tokens). Deps + file paths = ~1K chars (~250 tokens). System prompt = ~500 tokens. Total input: ~10K tokens. Output: ~500 tokens for 5 items. Well within GPT-5.4-mini's context window. Cost is covered by the user's ChatGPT/Codex subscription — no per-call API billing needed.

### Output Format

**Design decisions (from /plan-design-review):**
- Header leads with the payoff (count + % ignored), not the date
- Tags use semantic colors: `[bold green]ADOPT[/bold green]`, `[bold red]AVOID[/bold red]`, `[bold cyan]COPY[/bold cyan]`
- No engagement metrics in digest (relevance score already filtered; metrics are noise here)
- No horizontal rule separators (blank lines between items, matching `xx feed` pattern)
- Numbered indices for scannability and consistency with `xx feed`
- Author format matches `format.py`: `[bold]{name}[/bold] [dim]@{username} · {time}[/dim]`
- Tweet ID on its own dim line for follow-up actions
- Shared formatting helpers: extract `_format_author()` from `print_tweet()`, reuse in `print_digest_item()`
- Content width capped at `min(console.width - 4, 88)` for tweet text and explanation wrapping
- Minimum supported terminal width: 60 columns. Graceful degradation below that.
- Display names truncated to 20 chars with ellipsis in digest view (username is the reliable identifier)
- `NO_COLOR` env var respected (Rich handles natively). Bold text on tags ensures minimum differentiation without color.
- When piped (`not console.is_terminal`): no spinner, no colors, no footer interaction hint

**Rich markup specification:**
- Header line: `[bold]xx digest[/bold] [dim]—[/dim] [bold]{count}[/bold] signals`
- Subheader: `{ignored_pct}% of your timeline was noise.`
- Context line: `[dim]{repo} ({summary}) · {scanned} scanned · {date}[/dim]`
- Tag: `[bold green]ADOPT[/bold green]` / `[bold red]AVOID[/bold red]` / `[bold cyan]COPY[/bold cyan]`
- Author: `[bold]{display_name}[/bold] [dim]@{username} · {relative_time}[/dim]`
- Tweet text: plain, 2-space indent
- Why line: `[dim]Why:[/dim] {explanation}`, 2-space indent
- Tweet ID: `[dim]id:{tweet_id}[/dim]`, 2-space indent
- Footer: `[dim]Run xx like <id> or xx reply <id> "text" to interact.[/dim]`

### Resolved Design Decisions (from /plan-design-review, Pass 7)

- **Relevance score visible:** Yes, as `[dim]{score}/10[/dim]` on the ID line. Low visual weight, available for calibration.
- **Serendipity item treatment:** Subtle `[dim](wildcard)[/dim]` label after the tag. Same format as regular items. The label explains why it's there without diminishing it.
- **Footer hint suppression:** Deferred to v2. Suppress after N runs or via config flag.
- **`--brief` mode:** Deferred to v2. Suppresses "Why" explanations for power users who scan by tag + author only.

### NOT in scope (design review)

Design decisions explicitly deferred, not forgotten:
- **Dark/light theme-aware colors** — `bold` tags provide minimum differentiation. Full theme support is v2.
- **`--brief` flag for repeat users** — the "Why" explanation is v1's core value prop. Suppression is premature.
- **Footer hint auto-suppression** — track run count to suppress after 3-5 runs. Requires state tracking, defer to v2.
- **Multi-repo context display** — header shows one repo. If multi-repo is added in v2, header needs redesign.
- **Digest history/diff** — "3 new signals since your last digest." Requires state. v2.

Existing patterns in `format.py` that the digest MUST reuse:

### First-Run Setup (from /plan-design-review)

When no `default_repo` is configured and the current directory is not a git repo, the digest triggers an interactive setup before proceeding:

1. Pull timeline first (so tweets are ready when context is built)
2. Detect common dev directories (`~/Code/`, `~/Projects/`, `~/Developer/`, `~/repos/`)
3. List git repos found, numbered for selection
4. User picks one (or enters a custom path)
5. Save to `~/.xxcli/config.yaml` as `default_repo`
6. Continue with digest using the selected repo as context

```
xx digest

⏳ Pulling timeline...

No work context configured yet.
Which repo are you working on?

Found git repos in ~/Code/:
  1. xxcli
  2. my-saas-app
  3. dotfiles

Select repo [1]: 1
✔ Saved ~/Code/xxcli as default repo.

⏳ Building work context from xxcli...
⏳ Scoring 47 tweets against your work context...

[digest renders with context]
```

If no dev directories found or no git repos in them, fall back to: `Enter repo path (or press Enter to skip):`. If skipped, run without context and show the inline warning.

### Scheduling (cron-friendly)

The digest command is stateless and cron-friendly:
```bash
# Run daily at 9am
0 9 * * * cd ~/Code/xxcli && xx digest --repo ~/Code/current-project >> ~/.xxcli/digests/$(date +\%Y-\%m-\%d).txt
```

When piped (`not console.is_terminal`): suppress progress spinner, suppress footer interaction hint, let Rich auto-strip ANSI colors. The digest text content renders as clean plain text.

Future: `xx digest --schedule 9am` to manage cron entries directly.

### Error Handling

All errors use the existing `print_error()` from `format.py` (which prefixes `[red]Error:[/red]`).

- **Missing OpenAI API key:** `print_error("No OPENAI_API_KEY found. Set it in your environment (from your ChatGPT/Codex subscription).")` Exit 1. Check this BEFORE any API call — fail fast.
- **LLM API failure (rate limit, timeout, 500):** One retry with 3-second backoff. If retry fails, `print_error()` with the specific failure reason and suggest: "Try again in a few minutes, or check your API key/quota."
- **LLM returns malformed JSON:** With Pydantic structured output (see architecture), this error class is largely eliminated. If it still occurs, one retry with stricter prompt. If retry fails, `print_error()` with the raw response for debugging.
- **Twitter API failure:** Reuse the existing `_handle_api_error` pattern from `cli.py`. No special handling needed.
- **Zero tweets returned:** Skip LLM call entirely. Print: `"No tweets found in the last {period}. Your timeline may be empty or the API returned no results."`
- **Fewer than `--count` relevant tweets:** Show whatever scored >= 7, even if fewer than requested count. The header count ("1 signal") makes this clear. No apology needed.
- **`--repo` path is not a git repo:** On first run, trigger the interactive setup. On subsequent runs (or if `--repo` was explicitly passed), show inline warning and proceed without context.

### Configuration

All config values have hardcoded defaults. The config file (`~/.xxcli/config.yaml`) is entirely optional — the tool never requires it to exist. If present, it overrides defaults. `~/.xxcli/` directory is created on first use only if needed (e.g., when writing cron digest logs).

```yaml
default_repo: ~/Code/xxcli
default_since: 24h
default_count: 5
```

**Result: 1 high-relevance, 2 medium, 16 skip. ~5-10% signal rate.** The "3 items / 97% ignored" demo line was accurate.

### Key Learnings for Prompt Design

1. **Hype/reaction tweets are noise.** Tweets that just add excitement on top of news ("this is absolutely crazy") without expanding, challenging, or adding context should be classified as `skip`. The user wants perspectives that expand understanding, not amplify hype.

2. **"What can I do with this?" is the acid test.** The user's recurring filter was actionability — not "is this interesting?" but "does this change what I build or how I think about what I build?"

3. **Truncated tweets are a UX problem.** Several tweets were cut off ("for 36 year…", "real ways…"), making it impossible to judge relevance. The digest should fetch full tweet text, or at minimum flag when text is truncated.

4. **The serendipity constraint (CRITICAL).** User's own words: "the algo on top of the algo should not lead us down the most narrow black hole ever, or else that isn't good for new ideas and creativity since creativity is association and connection." The relevance filter must not be so tight that it kills serendipity. Prompt should include a small allowance for "adjacent interesting" — things that aren't directly about your current repo but expand your thinking in relevant directions. Maybe 1 out of 5 digest items can be a "wildcard" that scores 5-6 on direct relevance but high on novelty/surprise.

5. **RTs of truncated content are mostly noise.** Several tweets were RTs where the interesting content was cut off. Pre-filtering or de-prioritizing RTs (unless the RT adds commentary) would improve signal.

### Using This Eval

When iterating on prompt quality (Next Steps, step 6), run the digest against these 19 tweets and compare its picks against the hand-labeled verdicts above. The digest should:
- Surface tweet #5 (Stripe Projects) as HIGH / `adopt` or `copy`
- Surface tweet #6 or #8 (Codex) as MEDIUM, ideally noting the iOS caveat for #6
- Skip tweets #1, #3, #7, #10, #11, #13 with high confidence
- The hard cases are #9 (reaction tweet) and #19 (tangentially related) — getting these right means the prompt understands the difference between "interesting" and "useful"

## First-Run Eval Wizard (NEEDS /plan-ceo-review)

**Status:** Idea captured. Run /plan-ceo-review on this section before designing or building.

**Core insight:** The onboarding experience IS the eval. Instead of a cold setup that asks "which repo?" and then guesses what you care about, the first run walks you through a calibration session that simultaneously:
1. Learns what you're working on (repos, goals, current focus)
2. Builds a labeled eval dataset from your actual preferences
3. Personalizes the digest prompt using your real judgments

The hand-labeled eval in `evals/xx-feed-20260327.md` is exactly this process done manually. The wizard automates it.

### The Flow (sketch, not designed)

**Phase 1: What are you working on?**
- Scan common dev directories (`~/Code/`, `~/Projects/`, etc.) for git repos
- Present them as a numbered list, user picks their primary repo
- Also ask: "Anything else that captures what you're working on?" — could be other repos, notes files, project docs
- The repo is the primary context source. Other pointers are supplementary.
- Save selections to `~/.xxcli/config.yaml`

**Phase 2: What matters to you?**
- Ask high-level preference questions: "What topics are most relevant to your work right now?" (AI/ML, web dev, mobile, infra, design, business, etc.)
- Ask: "What kind of content is noise for you?" (hype tweets, reaction tweets, memes, news without analysis, etc.)
- These preferences shape the LLM's system prompt and scoring weights

**Phase 3: Calibration eval (the key innovation)**
- Pull real tweets from the user's timeline (same API call as a normal digest)
- Walk through 10-15 tweets one by one using interactive prompts
- For each tweet, ask the user: "Is this useful to you right now?"
  - Options: Useful / Maybe / Skip / Noise
  - Optionally: "Why?" (free text, helps tune the prompt)
- This creates a labeled dataset stored in `~/.xxcli/evals/` or `evals/`
- The labeled data serves two purposes:
  1. **Eval ground truth** — run the digest scoring against these labels to measure prompt quality
  2. **Prompt personalization** — include examples of "what this user considers useful" in the LLM prompt as few-shot examples

**Phase 4: First digest**
- Run the digest with the calibrated context and preferences
- Show the results with the established output format
- The user immediately sees whether the calibration worked

### Why This Matters

The existing eval in `evals/xx-feed-20260327.md` proved the concept: hand-labeling 19 tweets produced sharp insights about what the prompt should optimize for (hype is noise, "what can I do with this?" is the acid test, serendipity matters). But it was a manual process done in a markdown file. The wizard makes this a product feature.

This is agentic — the tool guides the user through a structured conversation, learns from their responses, and immediately applies what it learned. It's not a config form. It's a calibration session.

### Open Design Questions (for /plan-ceo-review)

1. **Scope for v1:** Is this the first-run experience, or a separate `xx setup` / `xx calibrate` command? Should it block the first digest, or be optional?
2. **Eval storage:** Where do labeled tweets live? `~/.xxcli/evals/` (local, private) or `evals/` in the repo (shareable, version-controlled)?
3. **Re-calibration:** Can the user re-run the wizard to update preferences? How does that interact with existing eval data?
4. **How many tweets to calibrate on?** 10 is quick but may not cover enough variety. 20 takes too long for a setup flow. The eval data showed ~5-10% signal rate, so 15-20 tweets gives 1-2 positive examples.
5. **Few-shot vs system prompt:** Should calibration examples go in the LLM prompt as few-shot demonstrations, or distill into preference rules in the system prompt? Few-shot is more accurate but costs more tokens.
6. **The "email" angle:** Could the eval results or digest output be formatted and sent as an email digest? Or was "email" shorthand for "create the eval for you"?

## Open Questions

1. **Tweet deduplication.** Home timeline can include retweets and quote tweets of the same content. Should the digest deduplicate before sending to the LLM, or let the LLM handle it? Leaning toward pre-dedup (strip RTs of already-seen tweet IDs) to save tokens.
2. **Digest history / seen-items cache.** Should the tool track which tweets it has already surfaced to avoid repeats across runs? v1: no cache, keep it stateless. Add cache in v2 if repeats are a problem.
3. **Multiple repo context.** The `--repo` flag takes one path. v1: one repo. If user wants multiple, they run the digest multiple times or we add multi-repo in v2.
4. **Serendipity budget.** How many of the N digest items should be "wildcards" — not directly relevant to current work but high on novelty/surprise? Proposal: 1 out of 5 items can be a wildcard (relevance 5-6 but novelty 8+). Needs tuning against real usage.

## Success Criteria

1. Running `xx digest` produces a useful, non-generic summary that references your actual work context.
2. At least 1 out of 5 surfaced items leads to a concrete action (clicking a link, liking, replying, or investigating further).
3. The user runs `xx digest` instead of opening X in the browser for at least 3 consecutive days.
4. Total digest generation time is under 30 seconds (API calls + LLM processing).
5. Digest runs within the user's existing ChatGPT/Codex subscription limits with no additional API costs.
